# Private Chat Wire Protocol

This document specifies the on-wire encoding of the two packet types that
implement 1-to-1 private messaging in the Rivr mesh.

---

## Packet types

| Type | Name                   | Value | Direction          |
|------|------------------------|-------|--------------------|
| 12   | `PKT_PRIVATE_CHAT`     | 0x0C  | Unicast source→dst |
| 13   | `PKT_DELIVERY_RECEIPT` | 0x0D  | Unicast dst→source |

Both types are carried inside the standard Rivr frame header (see
[ARCHITECTURE.md](../ARCHITECTURE.md)).  The maximum payload allowed per
frame is 230 bytes (`RIVR_PKT_MAX_PAYLOAD`).

---

## PKT_PRIVATE_CHAT payload (type 12)

Maximum payload: 229 bytes (29-byte header + up to 200-byte body).

```
Offset  Width  Type    Field
──────  ─────  ──────  ───────────────────────────────────────────────────────
  0..7    8    u64 LE  msg_id           Stable unique ID: (src_id << 32) | sender_seq
  8..11   4    u32 LE  sender_seq       Per-sender monotonic counter (wraps at 2³²)
 12..15   4    u32 LE  timestamp_s      Sender Unix time in seconds (0 = unknown)
 16..17   2    u16 LE  flags            PCHAT_FLAG_* bitmask (see §Flags)
 18..19   2    u16 LE  expires_delta_s  0 = no expiry; else secs after timestamp_s
 20..27   8    u64 LE  reply_to_msg_id  0 = not a reply; else original msg_id
   [28]   1    u8      body_len         Byte count of UTF-8 body (0..200)
 29..N    N    UTF-8   body             Message text (N = body_len)
```

### msg_id derivation

```c
msg_id = ((uint64_t)src_id << 32) | sender_seq;
```

The ID is deterministic and requires no random number generator.  It is
stable across retransmits; the `PCHAT_FLAG_RETRANSMIT` flag distinguishes
re-sends from initial sends.

### Flags (u16 bitmask)

| Bit | Constant                  | Meaning                                      |
|-----|---------------------------|----------------------------------------------|
| 0   | `PCHAT_FLAG_ACK_REQUIRED` | Request wire-level ACK from the next hop     |
| 1   | `PCHAT_FLAG_RECEIPT_REQ`  | Request end-to-end delivery receipt          |
| 2   | `PCHAT_FLAG_RETRANSMIT`   | This frame is a retransmit of the same msg   |
| 3   | `PCHAT_FLAG_STORE_FWD_OK` | Intermediate relay may buffer the frame      |
| 4   | `PCHAT_FLAG_USER_VISIBLE` | Should surface in UI                         |
| 5   | `PCHAT_FLAG_SYSTEM_GENERTED` | Generated by firmware, not the user       |

Default flags for user-originated messages:

```
PCHAT_FLAG_ACK_REQUIRED | PCHAT_FLAG_RECEIPT_REQ | PCHAT_FLAG_USER_VISIBLE
```

## PKT_DELIVERY_RECEIPT payload (type 13)

Fixed 17 bytes.

```
Offset  Width  Type    Field
──────  ─────  ──────  ────────────────────────────────────────────
  0..7    8    u64 LE  orig_msg_id   msg_id of the message being acknowledged
  8..11   4    u32 LE  sender_id     Node ID of the original message sender
 12..15   4    u32 LE  timestamp_s   Receipt generation time (0 = unknown)
   [16]   1    u8      status        RCPT_STATUS_* code (see §Receipt status codes)
```

### Receipt status codes

| Value | Constant                  | Meaning                                              |
|-------|---------------------------|------------------------------------------------------|
| 0x00  | `RCPT_STATUS_DELIVERED`   | Message accepted by the application at destination   |
| 0x01  | `RCPT_STATUS_REJECTED`    | Destination rejected (e.g. spam or queue full)       |
| 0x02  | `RCPT_STATUS_EXPIRED`     | Message expired at destination before delivery       |
| 0x03  | `RCPT_STATUS_UNSUPPORTED` | Destination node does not support private chat       |

---

## RX deduplication

The receiving engine maintains a 32-slot (`PRIVATE_CHAT_DEDUP_SIZE`) LRU cache
of `(msg_id, from_id)` pairs.  A packet is dropped as a duplicate if the same
pair has been seen within the cache window.  Using `from_id` as part of the key
prevents a malicious node from suppressing delivery by spoofing `msg_id`.

---

## Receipt rate limiting

To prevent receipt storms on a lossy mesh, each node limits outgoing receipts
to **4 receipts per 10 s window** (`PRIVATE_CHAT_RECEIPT_RATE_MAX` per
`PRIVATE_CHAT_RECEIPT_RATE_WIN_MS`).  Excess receipts are silently dropped at
the sender; the originator's receipt-timeout (`RECEIPT_TMO_MS = 60 s`) will
expire to `DELIVERY_UNCONFIRMED` instead of `DELIVERED` in this case.

---

## Routing

`PKT_PRIVATE_CHAT` and `PKT_DELIVERY_RECEIPT` are unicast.  The sender looks
up the next hop in `route_cache`; if no route is found, a `ROUTE_REQUEST` is
emitted and the message is held as `AWAITING_ROUTE` for up to
`PRIVATE_CHAT_QUEUE_EXPIRY_MS` (30 s).

Relay nodes forward unicast frames to the next hop without inspecting the
private-chat payload.  The payload is **not encrypted** in this release; the
network-layer src/dst IDs provide addressing only.
